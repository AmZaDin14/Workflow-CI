## Why the CI workflow doesn't run `docker build` and `docker push` itself

The GH Actions runner's Docker daemon returns `malformed HTTP Authorization header`
when authenticating against Docker Hub with our PAT (a known docker/login-action
quirk with shorter-format PATs). The image is therefore built and pushed from
your local environment using the same Docker Hub credentials.

The CI workflow (`ci.yml`) does the rest end-to-end:
  - sets up Python 3.12
  - runs the MLflow Project entry point (`modelling.py`)
  - uploads the model artifact (`mlflow-model`)
  - verifies the image is publicly available on Docker Hub

To rebuild and push the image locally:
  # 1) Build with mlflow
  mlflow models build-docker -m runs:/<run_id>/model -n amzadin14/smsml-adult-mlflow

  # 2) (optional) build the slim variant for fast serving
  cd K3_workflow_ci/MLProject
  cp -r <mlflow_run_artifacts>/model ./model
  docker build -t amzadin14/smsml-adult:latest .

  # 3) Push both
  docker push amzadin14/smsml-adult-mlflow:latest
  docker push amzadin14/smsml-adult:latest

To run the model:
  docker run --rm -p 8080:8080 amzadin14/smsml-adult-mlflow:latest
  # OR (slim variant):
  docker run --rm -p 8080:8080 amzadin14/smsml-adult:latest

  # In another shell:
  curl -s -X POST http://localhost:8080/invocations -H "Content-Type: application/json" -d '{
    "dataframe_split": {
      "columns": ["age","education-num","capital-gain","capital-loss","hours-per-week","workclass","marital-status","occupation","relationship","race","sex","native-country"],
      "data": [[25,13,0,0,40,"State-gov","Never-married","Adm-clerical","Not-in-family","White","Male","United-States"]]
    }
  }'
  # {"predictions": [0]}
